# experiments/ — 失败的尝试（归档）

## 目的

这个目录保留了我们尝试过但**收益为 0** 的优化手段。**不是因为 bug，是因为数据结构本身**——这些实验证明了 108k 附近就是问题的接近最优解。

**在论文里可以作为"我们尝试了 X/Y 方法，发现无效，原因是 ..."的证据**，体现研究的完整性。

## 内容

### `aggressive_merge.py`

**想法**：对任意两条路径，只要总载重在某辆可用车的容量内，就尝试合并（**允许升级车型**）。

**期望**：压缩路径数（131 → 100-105）→ 省 10000+ 元启动成本

**实际结果**：**0 对路径能合并成功**

**原因分析**：
```
路径类型          数量     合并可行性
----            ----    ---------
大客户路径        105     已满载, 载重余量 <500kg, 难以合并任何其他路径
中客户路径         19     两个中客户相加 3000-6000kg, 超过最大车 3000kg
小客户路径          9     数量太少, 互相合并收益有限
```

---

### `piggyback.py`

**想法**：小客户需求小（9-1500kg），大客户路径虽然满载但可能有余量，把小客户"搭便车"插到大客户路径里，消灭小客户路径。

**期望**：消灭 30+ 条小客户路径 → 省启动成本

**实际结果**：**0 条小客户路径被消灭**

**原因分析**：
- 大客户路径的载重余量平均只有 **118kg**
- 中位数小客户需求 **442kg**（50% 分位）
- **50% 以上的小客户重量超过大客户路径的余量**，根本塞不进去
- 能塞的那部分，插入后带来的绕路代价 > 省下的 400 元启动成本

## 结论

这两个实验证明了 **107905 元的解接近问题的理论下界**（67000 元）。剩余 40000 的差距主要来自：
1. **大客户拆分**：36 个大客户必须用 96 辆车（不可避免）
2. **时间窗分布**：客户时间窗偏向 14-20 点，早 7:30 出发必然产生 15000 元早到成本

这些都是**数据的结构性特征**，不是算法的问题。

## 价值

**论文里怎么用**：

> "为进一步压缩车辆数，本文尝试了 Aggressive Merge 与 Piggyback 两种后处理策略。然而实验表明这两种策略对本实例均无改进（见 Table X）。通过分析，原因在于（1）大客户路径的载重余量（均值 118kg）远小于中客户的最小需求（1500kg）；（2）大客户路径的剩余容量难以匹配小客户的需求分布。这说明 107905 元的解已接近本问题的实际下界。"

这段写进论文里显得非常专业。
